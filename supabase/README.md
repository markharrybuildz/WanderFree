# supabase

Database schema for WanderFree. Single Postgres instance hosted on Supabase,
with Auth and Row Level Security doing the per-user filtering so the mobile
app can talk to Postgres directly without a backend API.

The catalog is curated by hand in the Supabase dashboard. The repo holds the
migrations (generated by `supabase db pull`) so the schema is reproducible.

## Layout

```
supabase/
├── migrations/      Timestamped SQL migrations. Regenerate with
│                    `supabase db pull` after editing the live DB.
└── README.md
```

## Schema overview

Two groups of tables.

**Catalog (public-read, hand-curated):**

```
card_issuers ──┐
               ↓
        card_products ──→ rewards_programs ──→ program_transfer_partners ──→ transfer_partners
               │
               ├──→ benefit_definitions ──→ benefit_categories
               │
               └──→ card_reward_rules ───→ reward_categories (self-referential parent_id)
```

**Per-portfolio (RLS-gated):**

```
profiles  ──→ portfolio_members ←── portfolios ──→ wallet_accounts ──→ rewards_programs
                                         │
                                         ↓
                                    user_cards ──→ card_products
                                         │
                          ┌──────────────┼──────────────────┐
                          ↓              ↓                  ↓
                user_signup_bonuses  spend_entries  user_benefit_cycles
                          │              │                  │
                          └──────────────┘                  ↓
                                                  benefit_redemptions
```

Key things to know:

- **Portfolios are shared.** A portfolio can have multiple `portfolio_members` (each with a `role`). All per-user data hangs off `portfolio_id`, not `auth.uid()`.
- **Reward earn rules vs. benefits.** `card_reward_rules` model multipliers on spend categories. `benefit_definitions` model recurring credits / perks. Don't conflate them.
- **Benefit cycles** instantiate a `benefit_definition` per `user_card` per period. `benefit_redemptions` hang off a specific cycle.

## Running migrations

For local development, install the Supabase CLI:

```bash
brew install supabase/tap/supabase

supabase login
supabase link --project-ref <your-project-ref>

# Pull the live schema into a fresh migration file. Use this whenever the
# DB has been edited via the Supabase dashboard.
supabase db pull

# Apply pending migrations to the linked remote project
supabase db push

# Or for a fresh local Postgres started by `supabase start`:
supabase migration up
```

Migrations are timestamped and applied in order. To create a new empty one:

```bash
supabase migration new <name>
```

## RLS principle

Catalog tables (`card_issuers`, `card_products`, `rewards_programs`,
`reward_categories`, `benefit_categories`, `benefit_definitions`,
`card_reward_rules`, `transfer_partners`, `program_transfer_partners`) all
have a `catalog read` policy: `SELECT` to `authenticated`, no INSERT/UPDATE/
DELETE. Writes go through the Supabase dashboard (service role).

Per-portfolio access is gated by two SQL helper functions defined in the
migration:

- `is_portfolio_member(portfolio_id uuid)` — true when the calling user
  appears in `portfolio_members` for that portfolio.
- `can_access_user_card(user_card_id uuid)` — true when the calling user is
  a member of the portfolio that owns the card. Used by every table that
  hangs off `user_cards` (`user_benefit_cycles`, `benefit_redemptions`,
  `user_signup_bonuses`, `spend_entries`).

Per-table summary:

| Table | Read | Write |
|-------|------|-------|
| `profiles` | own row only | own row only (update; no delete) |
| `portfolios` | members + creator | INSERT: anyone (sets `created_by = auth.uid()`); UPDATE/DELETE: creator only |
| `portfolio_members` | own row or other members of shared portfolios | gated by portfolio membership |
| `wallet_accounts`, `user_cards` | portfolio members | portfolio members |
| `user_benefit_cycles`, `benefit_redemptions`, `user_signup_bonuses`, `spend_entries` | `can_access_user_card(user_card_id)` | same |

The **service role key** bypasses RLS and must never ship in the mobile app.
